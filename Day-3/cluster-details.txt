 Scenario: kubeadm join command is missing / lost
❓ “I set up an on-prem cluster. Later, the kubeadm join command is missing. How do I add new worker nodes? What happens to existing nodes?”
✅ Short Answer
Already joined nodes → NOTHING happens (they keep working)


New nodes → You must generate a new join command


Cluster state is NOT lost



🔹 Where kubeadm Stores Information
📌 Worker Node
/etc/kubernetes/kubelet.conf

What it contains
Client certificate


API server endpoint


Used by kubelet to authenticate


✔ This is why already joined nodes do NOT need to rejoin

🔹 What Happens If Server Restarts?
🔁 Worker Node Restart
kubelet starts


reads /etc/kubernetes/kubelet.conf


reconnects to API server


✅ No re-join required

🔁 Control Plane Restart
kubelet starts


static pods restart automatically:


kube-apiserver


etcd


controller-manager


scheduler


✅ Cluster recovers automatically

❌ When Re-Join Is Required?
Only if:
/etc/kubernetes is deleted


Node OS reinstalled


Certificates are manually removed

🔹 kubeadm init – What It REALLY Does
kubeadm init

🔹 Internal Steps
1️⃣ Generates CA & certificates
 2️⃣ Starts etcd
 3️⃣ Starts API server
 4️⃣ Starts controller-manager & scheduler
 5️⃣ Writes kubeconfigs
 6️⃣ Creates static pod manifests
 7️⃣ Generates bootstrap token
 8️⃣ Prints kubeadm join command
📌 Files Created
/etc/kubernetes/pki/
/etc/kubernetes/manifests/
/etc/kubernetes/admin.conf

⚠️ Important
❌ kubeadm init should be run ONLY ONCE

2️⃣ Token

➡️ Temporary authentication token

3️⃣ CA Hash

➡️ Prevents MITM attack
 ➡️ Confirms API server identity

🔹 What kubeadm join Does Internally
1️⃣ Verifies token
 2️⃣ Validates API server certificate
 3️⃣ Fetches cluster CA
 4️⃣ Generates kubelet cert
 5️⃣ Writes /etc/kubernetes/kubelet.conf
 6️⃣ Registers node in API server

🔹 If kubeadm token expires, what will happen?
👉 Nothing breaks in your existing cluster.
 This is a very common misunderstanding, so let’s be very clear.

✅ What DOES NOT happen
If the token expires:
❌ Existing worker nodes do NOT go NotReady
 ❌ Running pods do NOT stop
 ❌ kubelet does NOT disconnect
 ❌ Control plane does NOT restart
 ❌ Cluster does NOT lose state
The token is NOT used for daily authentication.

❌ What DOES happen
🚫 New nodes cannot join the cluster using the expired token
You’ll see errors like:
token has expired

or
authentication failed

That’s it.

🔹 Why Existing Nodes Keep Working (Important)
Once a node joins:
kubeadm creates certificates


kubelet uses:

 /etc/kubernetes/kubelet.conf


kubelet authenticates using client certificates, not tokens


➡️ Tokens are one-time bootstrap only.

🔹 Token Usage Lifecycle
kubeadm join
   ↓
token used ONCE
   ↓
kubelet cert issued
   ↓
token no longer needed


🔹 Token Expiry Default
Default TTL: 24 hours


Stored in:

 kube-system namespace


Check tokens:
kubeadm token list


🔹 How to Fix Token Expiry (Safe & Easy)
On control plane:
kubeadm token create --print-join-command

✔ Generates new token
 ✔ Prints new join command
 ✔ Existing nodes unaffected

🔹 Create a Token That Never Expires (Not Recommended for Prod)
kubeadm token create --ttl 0

⚠️ Use only in labs
